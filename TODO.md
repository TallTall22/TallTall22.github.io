> **Framework**: Vue 3 `<script setup>` + TypeScript strict (zero `any`) + Vuetify 3 + Pinia Setup Store  
> **Generated by**: BDD Requirements Expert (Recursive BDD-Atomic Framework)  
> **Feature**: F-08.1 ~ F-08.5 — Trip timeline rendering, per-date itinerary cards, matchup format, local time + timezone, scrollable strip

## Executive Summary

**What**: Render a horizontally-scrollable timeline strip below the map/control-panel area in `App.vue`. Each day of `tripStore.selectedTrip.itinerary` is represented as a card: game days show the matchup label (`Away Nickname @ Home Nickname`), local start time with abbreviated timezone (e.g. `7:05 PM ET`), stadium name, and team logos; travel days show a simplified "travel" card with optional distance. When no trip exists, an empty-state placeholder is shown.

**Why**: Users need a linear, date-ordered view of their generated itinerary to understand sequencing, travel days, and game times at a glance — complementing the spatial map view.

**Scope Boundaries**:
- ✅ IN: `TripTimelineCard.vue` — single day card (Atom/Molecule)
- ✅ IN: `TripTimelineStrip.vue` — horizontally scrollable container (Organism)
- ✅ IN: `useTimeline` composable mapping `TripDay[]` → `TimelineDayViewModel[]`
- ✅ IN: Empty state (no trip), travel day variant, game day variant
- ✅ IN: `teamId` → `teamNickname` mapping via stadiums list (using `Stadium.teamId` NOT `Stadium.id`)
- ✅ IN: Local time + IANA timezone → abbreviated name via `Intl.DateTimeFormat`
- ✅ IN: Integration in `App.vue` as a full-width row below the two-column layout
- ✅ IN: Race condition guard (`requestCounter` + `isMounted`) matching project pattern
- ❌ OUT: Editing/reordering itinerary days (read-only display)
- ❌ OUT: Click-to-navigate to game detail (future feature)
- ❌ OUT: Vertical scroll mode (horizontal only for F-08.5)
- ❌ OUT: Server-side rendering / hydration (client-only SPA)

## Domain Analysis & Recursive Validation

### Given-When-Then Scenarios (Behavioral Spec)

---

**Scenario 1: No trip generated yet — empty state**

- **Given**: App has mounted; `tripStore.selectedTrip` is `null`; user has not yet confirmed a date range
- **When**: `TripTimelineStrip` mounts and reads `useTimeline`
- **Then**:
  - `timelineDays` computed returns `[]`
  - `TripTimelineStrip` renders an empty-state placeholder (e.g. "Generate a trip to see your itinerary")
  - No cards are rendered; no loading spinner is shown
  - Strip does not reserve excessive vertical space (collapsed or minimal-height placeholder)

---

**Scenario 2: Trip with mixed game days and travel days — happy path**

- **Given**: `tripStore.selectedTrip` is set to a `Trip` with 5 days: `[GameDay, TravelDay, GameDay, GameDay, TravelDay]`; `loadStadiums()` resolves successfully with all 30 stadiums
- **When**: `useTimeline` watch fires and `buildTimelineDays()` executes
- **Then**:
  - `timelineDays.value` has exactly 5 `TimelineDayViewModel` entries
  - GameDay entries have `type: 'game_day'`, `matchupLabel: 'Away Nickname @ Home Nickname'`, `localTime: '7:05 PM ET'`
  - TravelDay entries have `type: 'travel_day'`, `matchupLabel: null`, `localTime: null`
  - `TripTimelineStrip` renders 5 cards in a horizontally scrollable row
  - Game day cards show home/away team logos; travel day cards show a travel icon

---

**Scenario 3: `teamId` not found in stadiums — graceful fallback**

- **Given**: A `GameDay` has `game.homeTeamId = '999'` (unknown team); stadiums list does not contain a stadium with `teamId = '999'`
- **When**: `useTimeline` maps that `GameDay` to a `TimelineDayViewModel`
- **Then**:
  - `matchupLabel` falls back to `'Unknown @ Unknown'` (or uses available side if only one is missing)
  - `homeTeamLogo` and/or `awayTeamLogo` fall back to empty string `''`
  - No thrown error; no silent `undefined` leak; card still renders with fallback content
  - `isLoading` returns to `false`; `error` remains `null` (partial data fallback, not an error state)

---

**Scenario 4: Local time + timezone abbreviation rendering**

- **Given**: A `GameDay` stadium has `timeZone: 'America/Chicago'` (CDT/CST); `game.startTimeLocal = '6:05 PM'`
- **When**: `useTimeline` formats the time display for that day
- **Then**:
  - `localTime` is `'6:05 PM CDT'` (or `'6:05 PM CST'` depending on DST at the game date)
  - The timezone abbreviation is derived via `Intl.DateTimeFormat` with `timeZoneName: 'short'` — NOT hardcoded
  - Rendering uses `startTimeLocal` from game data for HH:MM display + extracted abbreviation from `Intl` formatter

---

**Scenario 5: `loadStadiums()` fails — error state**

- **Given**: `stadiumService.loadStadiums()` returns `{ stadiums: [], error: 'FETCH_FAILED' }`; a trip is set in the store
- **When**: `useTimeline` attempts to build timeline days
- **Then**:
  - `isLoading.value` becomes `false`
  - `error.value` is set to a user-readable string (e.g. `'Stadium data unavailable'`)
  - `timelineDays.value` returns `[]`
  - `TripTimelineStrip` renders the error state variant (e.g. error alert or fallback message) instead of cards

---

**Scenario 6: Rapid trip regeneration — race condition guard**

- **Given**: User clicks "Generate Trip" twice in quick succession; two async `buildTimelineDays` calls fire (request IDs 1 and 2)
- **When**: Both calls await `loadStadiums()` (which is cached so resolves synchronously on second call); request 2 completes after request 1
- **Then**:
  - Only the result from the **latest** request (ID 2) is committed to `timelineDays.value`
  - Request 1 result is discarded via `requestCounter` guard (same pattern as `useMapPolylines.ts`)
  - No stale partial state is rendered

---

**Scenario 7: Trip set then cleared (trip → null transition)**

- **Given**: `tripStore.selectedTrip` was previously a valid `Trip` (5 days, rendered correctly)
- **When**: `store.setSelectedTrip(null)` is called (user resets, or new generation starts)
- **Then**:
  - `timelineDays.value` immediately resets to `[]`
  - `TripTimelineStrip` transitions back to empty state
  - No lingering card data from the previous trip

---

### Technical Feasibility Assessment

| Concern | Finding | Mitigation |
|---|---|---|
| **Race Condition** | `selectedTrip` watcher fires `loadStadiums()` (async); rapid trip regeneration risks stale commits | `requestCounter` + `isMounted` flag pattern (identical to `useMapPolylines.ts`) |
| **teamId Mapping** | `Game.homeTeamId`/`awayTeamId` are numeric strings matching `Stadium.teamId` — NOT `Stadium.id` (e.g. "NYY") | Build `Map<string, Stadium>` keyed by `Stadium.teamId` (not `Stadium.id`) in `useTimeline` |
| **Prop Drilling** | `TripTimelineStrip` → `TripTimelineCard` is only 1 level deep; no drilling concern | Direct prop passing; no Provide/Inject needed |
| **Type Safety** | `TripDay` is a discriminated union (`GameDay | TravelDay`); must narrow with `day.type === 'game_day'` before accessing `.game` | Use TypeScript discriminated union narrowing; `never` on `TravelDay.game` prevents accidental access |
| **Timezone Formatting** | `Intl.DateTimeFormat` with `timeZoneName: 'short'` returns full formatted string; need to extract only abbreviation | Use regex or `formatToParts()` to extract the `timeZoneName` part; avoid string parsing of `startTimeLocal` |
| **Performance** | `useTimeline` rebuilds on every `selectedTrip` change; 30-day itinerary = 30 ViewModels max | `computed()` memoization for `timelineDays`; stadium map built once per request, not per card |
| **Error Handling** | `loadStadiums()` never throws (returns `{ error }` pattern); teamId miss is partial fallback not error | Distinguish service-level errors (surface as `error` ref) from data-level misses (silent fallback in ViewModel) |
| **Scroll Behavior** | Horizontal scroll on desktop/mobile; touch support needed | CSS `overflow-x: auto` with `-webkit-overflow-scrolling: touch`; or Vuetify `v-slide-group` |
| **State Persistence** | Timeline should reset when trip resets; no persistence needed | `watch(selectedTrip, ...)` with `immediate: true` handles both null and value transitions |
| **Empty State UX** | Strip should not show loading on initial null state; only show loading when trip is set and stadiums are loading | Separate `isLoading` logic: skip loading state when `trip === null` |

```typescript
export function useTimeline(): UseTimelineReturn {
  const store = useTripStore();
  const { selectedTrip } = storeToRefs(store);

  const timelineDays = ref<TimelineDayViewModel[]>([]);
  const isLoading    = ref<boolean>(false);
  const error        = ref<string | null>(null);

  let requestCounter = 0;
  let isMounted      = true;
  onBeforeUnmount(() => { isMounted = false; });

  async function buildTimelineDays(requestId: number): Promise<void> { ... }

  watch(selectedTrip, () => {
    requestCounter++;
    void buildTimelineDays(requestCounter);
  }, { immediate: true });

  return { timelineDays, isLoading, error };
}
```

#### Timezone Abbreviation Pattern

```typescript
// Preferred approach: use formatToParts() to extract only the timezone part
export function extractTimeZoneAbbr(ianaTimeZone: string, referenceDate: Date): string {
  try {
    const parts = new Intl.DateTimeFormat('en-US', {
      timeZone:     ianaTimeZone,
      timeZoneName: 'short',
    }).formatToParts(referenceDate);
    return parts.find(p => p.type === 'timeZoneName')?.value ?? '';
  } catch {
    return '';
  }
}

// Combine with existing startTimeLocal from game data
export function formatLocalTime(
  startTimeLocal: string,   // e.g. "7:05 PM" — already formatted in game data
  ianaTimeZone:   string,   // e.g. "America/New_York"
  gameDate:       string,   // e.g. "2026-07-04" — used to get correct DST offset
): string {
  const refDate = new Date(`${gameDate}T12:00:00Z`); // noon UTC as reference
  const abbr = extractTimeZoneAbbr(ianaTimeZone, refDate);
  return abbr ? `${startTimeLocal} ${abbr}` : startTimeLocal;
}
```
